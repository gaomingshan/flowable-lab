## 第一迭代核心接口草案

> 修订说明：本文件的后续执行口径请同时参考 [第一迭代核心接口草案 1.1 修订](C:/soft/work/code/java/local/flowable-lab/doc/06-接口/第一迭代核心接口草案1.1-修订.md)。如与原文冲突，以修订说明为准。

### 1. 文档目标

本文档用于定义第一迭代最小可用的核心接口集合。

接口设计目标：

- 先支撑流程定义、流程绘制、版本、发布、启动、基础查询
- 不提前把外围配置和复杂运维纳入接口层
- 接口命名围绕平台语义，而不是 Flowable 原生语义

### 2. 接口分组

第一期建议分为六组接口：

1. 流程定义接口
2. 流程版本接口
3. 图模型与逻辑骨架接口
4. 节点候选人配置接口
5. 边条件配置接口
6. 流程发布与启动接口
7. 基础查询接口

### 3. 流程定义接口

### 3.1 创建流程定义

`POST /api/platform/workflow-definitions`

请求示例：

```json
{
  "definitionKey": "expense_approval",
  "definitionName": "报销审批",
  "categoryCode": "expense",
  "description": "报销审批流程"
}
```

响应关注点：

- definitionId
- 当前状态

### 3.2 查询流程定义列表

`GET /api/platform/workflow-definitions`

建议查询参数：

- `keyword`
- `categoryCode`
- `status`

### 3.3 查询流程定义详情

`GET /api/platform/workflow-definitions/{definitionId}`

返回：

- 定义基本信息
- 最新草稿版本号
- 最新发布版本号

### 3.4 修改流程定义基础信息

`PUT /api/platform/workflow-definitions/{definitionId}`

### 4. 流程版本接口

### 4.1 创建草稿版本

`POST /api/platform/workflow-definitions/{definitionId}/versions`

请求示例：

```json
{
  "changeNote": "新增并行审批分支"
}
```

### 4.2 查询版本列表

`GET /api/platform/workflow-definitions/{definitionId}/versions`

### 4.3 查询版本详情

`GET /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}`

返回：

- 图模型
- 节点边逻辑骨架摘要
- 校验结果

### 4.4 校验版本

`POST /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/validate`

返回：

- 图结构校验结果
- 节点边骨架校验结果
- 节点候选人和边条件配置校验结果

### 5. 图模型与逻辑骨架接口

### 5.1 查询版本图模型

`GET /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/graph`

返回：

- 原始 `graph_json`

### 5.2 保存版本图模型

`PUT /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/graph`

请求示例：

```json
{
  "graph": {
    "nodes": [],
    "edges": [],
    "viewport": {}
  }
}
```

说明：

- 后端保存 `graph_json`
- 后端同时解析并同步 `wf_node`、`wf_edge`

### 5.3 查询节点骨架列表

`GET /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/nodes`

### 5.4 查询边骨架列表

`GET /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/edges`

### 6. 节点候选人配置接口

### 6.1 查询节点候选人配置列表

`GET /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/nodes/{nodeKey}/candidate-configs`

返回示例：

```json
[
  {
    "id": "cfg_1",
    "nodeKey": "task_1",
    "policyType": "user",
    "policyMode": "single",
    "policyName": "固定审批人",
    "policyPayload": {
      "users": [
        {
          "userId": "u_1001",
          "userName": "张三"
        }
      ]
    }
  }
]
```

### 6.2 新增节点候选人配置

`POST /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/nodes/{nodeKey}/candidate-configs`

请求示例：

```json
{
  "policyType": "user",
  "policyMode": "candidate",
  "policyName": "财务候选人",
  "policyPayload": {
    "users": [
      {
        "userId": "u_1001",
        "userName": "张三"
      },
      {
        "userId": "u_1002",
        "userName": "李四"
      }
    ]
  }
}
```

### 6.3 修改节点候选人配置

`PUT /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/nodes/{nodeKey}/candidate-configs/{configId}`

### 6.4 删除节点候选人配置

`DELETE /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/nodes/{nodeKey}/candidate-configs/{configId}`

### 7. 边条件配置接口

### 7.1 查询边条件配置列表

`GET /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/edges/{edgeKey}/condition-configs`

### 7.2 新增边条件配置

`POST /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/edges/{edgeKey}/condition-configs`

请求示例：

```json
{
  "conditionName": "金额大于 1000",
  "conditionType": "expression",
  "expressionText": "${amount > 1000}",
  "isDefaultBranch": false
}
```

### 7.3 修改边条件配置

`PUT /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/edges/{edgeKey}/condition-configs/{configId}`

### 7.4 删除边条件配置

`DELETE /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/edges/{edgeKey}/condition-configs/{configId}`

### 8. 流程发布与启动接口

### 8.1 发布流程版本

`POST /api/platform/workflow-definitions/{definitionId}/versions/{versionNo}/release`

请求示例：

```json
{
  "releaseComment": "第一版上线"
}
```

返回关注点：

- releaseId
- releaseStatus
- deploymentId
- processDefinitionId

### 8.2 查询发布记录详情

`GET /api/platform/releases/{releaseId}`

### 8.3 查询允许启动流程列表

`GET /api/platform/launchable-workflows`

建议查询参数：

- `keyword`
- `categoryCode`

### 8.4 启动流程

`POST /api/platform/workflow-instances`

请求示例：

```json
{
  "definitionId": "1001",
  "businessKey": "EXP-20250001",
  "title": "张三报销单",
  "starterId": "u_0001",
  "starterDeptId": "dept_001",
  "variables": {
    "amount": 1000,
    "reason": "出差报销"
  }
}
```

返回关注点：

- instanceId
- engineProcessInstanceId
- currentNodeName
- status

### 9. 基础查询接口

### 9.1 查询待办列表

`GET /api/platform/query/todo`

建议查询参数：

- `assigneeId`
- `keyword`
- `pageNo`
- `pageSize`

### 9.2 查询已办列表

`GET /api/platform/query/done`

建议查询参数：

- `operatorId`
- `keyword`
- `pageNo`
- `pageSize`

### 9.3 查询我发起的流程列表

`GET /api/platform/query/initiated`

建议查询参数：

- `starterId`
- `keyword`
- `pageNo`
- `pageSize`

### 9.4 查询实例摘要

`GET /api/platform/query/instances/{instanceId}/summary`

返回关注点：

- 实例基础信息
- 当前节点
- 流程状态
- 发起人
- 发起部门
- 发布时间来源

### 10. 接口设计原则

第一期接口应遵循以下原则：

#### 10.1 以平台对象为中心

接口围绕：

- workflow definition
- workflow version
- graph
- node / edge logic
- node candidate config
- edge condition config
- release
- workflow instance
- query projection

而不是围绕 Flowable 原生对象直接暴露。

#### 10.2 图模型整份保存，外挂配置独立维护

图模型整份保存，但节点候选人和边条件配置独立维护，避免把外挂配置塞进 `graph_json`。

#### 10.3 查询接口基于平台读模型

第一期所有基础查询接口都应从平台投影读取，而不是直接拼接 Flowable 表。

### 11. 第一迭代不提供的接口

第一期不提供：

- 发布审批接口
- 运维动作接口
- 表达式目录接口
- 高级搜索接口
- 多实例/会签接口

### 12. 结论

这组接口已经足以支撑第一期原型：

- 定义流程
- 绘制并保存流程图
- 同步节点边逻辑骨架
- 维护节点候选人配置
- 维护边条件配置
- 发布到 Flowable
- 启动流程
- 查询基础可读模型

接口层同样保持了“平台主导、Flowable 收敛到引擎层”的设计原则。
