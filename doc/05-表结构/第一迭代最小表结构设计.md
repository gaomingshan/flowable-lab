## 第一迭代最小表结构设计

> 修订说明：本文件的后续执行口径请同时参考 [第一迭代最小表结构设计 1.1 修订](C:/soft/work/code/java/local/flowable-lab/doc/05-表结构/第一迭代最小表结构设计1.1-修订.md)。如与原文冲突，以修订说明为准。

### 1. 文档目标

本文档用于将第一迭代的平台领域模型收敛成真正可落地的最小表结构。

设计原则是：

- 只保留第一期主链路必须的表
- 图主存储继续使用 `graph_json`
- 节点和边单独落逻辑骨架表
- 所有外挂配置独立建表并直接挂载到节点或边
- 候选人动态数据统一使用 JSON 承载
- 先支撑定义、版本、发布、启动、基础查询

### 2. 第一迭代必须建的表

第一期建议落以下 13 张核心表：

1. `wf_definition`
2. `wf_definition_version`
3. `wf_node`
4. `wf_edge`
5. `wf_node_candidate_config`
6. `wf_edge_condition_config`
7. `wf_release`
8. `wf_engine_binding`
9. `wf_launchable_projection`
10. `wf_instance`
11. `wf_todo_projection`
12. `wf_done_projection`
13. `wf_initiated_projection`

这组核心表已经足以支撑：

- 流程定义
- 流程绘制
- 流程版本控制
- 流程发布
- 平台到 Flowable 的绑定
- 流程启动
- 基础查询

### 3. 表设计说明

### 3.1 `wf_definition`

用途：

- 表达一个长期存在的流程定义对象

建议字段：

| 字段 | 类型建议 | 说明 |
|---|---|---|
| `id` | bigint / varchar | 主键 |
| `tenant_id` | varchar | 租户标识 |
| `definition_key` | varchar | 流程定义唯一 key |
| `definition_name` | varchar | 流程定义名称 |
| `category_code` | varchar | 分类编码 |
| `description` | varchar / text | 描述 |
| `status` | varchar | 状态 |
| `latest_draft_version` | int | 最新草稿版本号 |
| `latest_released_version` | int | 最新发布版本号 |
| `owner_id` | varchar | 负责人 |
| `created_by` | varchar | 创建人 |
| `updated_by` | varchar | 更新人 |
| `created_at` | datetime | 创建时间 |
| `updated_at` | datetime | 更新时间 |

索引建议：

- `uk_definition_key(tenant_id, definition_key)`
- `idx_status(tenant_id, status)`

```json
{

      "userId": "u_1002",
      "userName": "李四"
```

### 4. 第一迭代不建的表

第一期建议明确不建以下表：

- `wf_release_approval`
- `wf_expression_catalog`
- `wf_process_variable_def`
- `wf_node_variable_def`
- `wf_constant_catalog`
- 各类高级搜索索引表

原因：

- 这些能力不影响第一期主链路
- 过早拆表会让原型过重

### 5. 建表顺序建议

建议按以下顺序建表：

1. `wf_definition`
2. `wf_definition_version`
3. `wf_node`
4. `wf_edge`
5. `wf_node_candidate_config`
6. `wf_edge_condition_config`
7. `wf_release`

10. `wf_instance`

12. `wf_done_projection`
13. `wf_initiated_projection`

### 6. 结论

这套最小表结构已经足够支撑第一期原型目标：

- 平台定义流程
- 平台保存版本和图 JSON
- 平台同步节点边逻辑骨架
- 平台独立维护节点候选人和边条件配置
- 平台发布并绑定 Flowable
- 平台提供启动入口
- 平台维护基础读模型

它故意没有把外围能力和后续治理能力做重，从而保证第一迭代可以轻装上阵。
