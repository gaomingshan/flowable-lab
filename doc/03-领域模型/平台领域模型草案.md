## 平台领域模型草案

> 修订说明：本文件的后续执行口径请同时参考 [平台领域模型草案 1.1 修订](C:/soft/work/code/java/local/flowable-lab/doc/03-领域模型/平台领域模型草案1.1-修订.md)。如与原文冲突，以修订说明为准。

### 1. 文档目标

本文档用于定义基于 Flowable 之上的平台层领域模型与表结构草案。

本次重点不是 Flowable 引擎内部实现，而是平台层最小可落地模型。

当前范围聚焦以下五类能力：

- 流程定义相关
- 流程发布相关
- 流程运维配置预留
- 外围业务配置预留
- 基础查询

同时遵循两个原则：

- 先保证最小迭代，不把外围配置做重
- 区分基础查询和高级查询

### 2. 总体原则

#### 2.1 平台模型是设计态主对象

平台拥有自己的：

- 流程定义
- 流程版本
- 流程图主存储
- 节点与边逻辑骨架
- 节点候选人配置
- 边条件配置
- 发布记录
- 查询投影

Flowable 只接收平台发布后的编译产物。

#### 2.2 图主存储与逻辑骨架并存

前端流程图仍然通过 `graph_json` 主存储。

同时平台内部保留：

- `wf_node`
- `wf_edge`

用于：

- 图结构校验
- 路径分析
- 配置挂载定位
- Flowable 模型转换

#### 2.3 外挂配置独立建模

第一期所有关键外挂配置都独立建表。

关系方向是：

- 候选人配置挂载到节点
- 条件配置挂载到边

而不是：

- 节点或边保存配置引用关系

#### 2.4 候选人动态数据使用 JSON 承载

第一期候选人配置主对象结构化建模，但策略明细使用 `policy_payload_json` 承载。

这样可以：

- 第一迭代先支持账号模式
- 第二迭代平滑扩展角色、部门、表达式模式
- 不推翻主表结构

#### 2.5 平台先做基础查询，高级查询后置

基础查询：

- 基于平台数据库读模型
- 只索引核心字段
- 服务于可发起流程、待办、已办、我发起的等基础页面

高级查询：

- 作为后续能力建设
- 基于中间件、消息投影、搜索索引实现
- 承接复杂筛选、全文搜索、组合维度查询

### 3. 领域拆分

平台当前建议拆成四个主要域和一个查询域。

#### 3.1 定义域

负责：

- 流程定义
- 流程版本
- 图 JSON 保存
- 节点边骨架维护
- 节点与边外挂配置管理

#### 3.2 发布域

负责：

- 校验
- 编译
- 部署到 Flowable
- 发布记录
- 引擎绑定关系

#### 3.3 运维配置预留域

第一期只保留扩展位置，不做复杂运维动作建模。

#### 3.4 外围语义配置预留域

第一期只落最小外挂配置，不做复杂外围能力细拆。

#### 3.5 查询域

负责：

- 可发起流程
- 我的待办
- 我的已办
- 我发起的流程
- 基础实例摘要

建议表：`wf_node_candidate_config`

建议字段：

- `id`

- `policy_type`
- `policy_mode`
- `policy_name`
- `policy_payload_json`
- `status`
- `created_by`
- `updated_by`
- `created_at`
- `updated_at`

说明：

- 节点候选人配置自己挂载到节点
- 第一迭代只支持账号模式
- `policy_payload_json` 统一承载策略明细

第一迭代账号模式示例：

```json
{
  "users": [
    {
      "userId": "u_1001",
      "userName": "张三"
    },
    {
      "userId": "u_1002",
      "userName": "李四"
    }
  ]
}
```

第二迭代可以平滑扩展为：

- 角色集合
- 部门集合
- 表达式配置

第一期优先落这些表：

- `wf_definition`
- `wf_definition_version`
- `wf_node`
- `wf_edge`
- `wf_node_candidate_config`
- `wf_edge_condition_config`
- `wf_release`

- `wf_instance`
- `wf_todo_projection`
- `wf_done_projection`
- `wf_initiated_projection`

### 6. 当前有意留白的部分

第一期先不深入：

- 高级搜索索引模型
- 表达式目录治理
- 角色、部门候选人治理
- 意见/附件分组治理
- 复杂变量体系
- 完整运维动作中心

### 7. 结论

平台层第一期最重要的不是建全所有外围模型，而是先把以下主链路打通：

- 定义
- 版本
- 发布
- 启动
- 基础查询

在这个基础上，再逐步向复杂流程能力、运维能力和高级搜索演进。
