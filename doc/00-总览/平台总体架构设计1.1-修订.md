## 平台总体架构设计 1.1 修订

### 1. 修订范围

本修订仅覆盖以下内容：

- 流程图主存储与节点边逻辑骨架的关系
- 外挂配置与节点边的挂载方式
- 候选人模块的一期建模方式

### 2. 修订内容

#### 2.1 流程图主存储仍然是 graph_json

修订口径：

- 前端流程图主存储仍然是 `wf_definition_version.graph_json`
- `graph_json` 是前端加载和保存的唯一主对象
- 第一迭代不单独引入 `wf_graph` 作为核心对象

#### 2.2 节点和边保留为逻辑骨架对象

修订口径：

- 节点和边必须保留为平台逻辑骨架对象
- 第一迭代落轻量结构化骨架表：
  - `wf_node`
  - `wf_edge`
- 这两张表只承载逻辑骨架字段，不承载复杂配置

#### 2.3 外挂配置独立建表，并直接挂载到节点或边

修订口径：

- 节点和边是流程逻辑骨架
- 外挂配置都是独立表
- 外挂配置自己声明挂载目标

关系方向是：

- 节点候选人配置挂载到节点
- 边条件配置挂载到边

而不是：

- 节点引用候选人配置
- 边引用条件配置

#### 2.4 候选人模块改为主配置结构化 + payload JSON

修订口径：

- 保留节点候选人主配置表 `wf_node_candidate_config`
- 不再拆账号明细表
- 候选人动态数据统一通过 `policy_payload_json` 承载

### 3. 当前执行口径

当前第一迭代研发应按以下口径执行：

1. `graph_json` 是流程图主存储
2. `wf_node`、`wf_edge` 是节点边逻辑骨架表
3. 节点和边不承载复杂配置
4. 外挂配置独立建表并直接挂载到节点或边
5. 候选人主配置结构化，动态策略数据通过 JSON 承载
