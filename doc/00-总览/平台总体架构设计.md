## 平台总体架构设计

> 修订说明：本文件的后续执行口径请同时参考 [平台总体架构设计 1.1 修订](C:/soft/work/code/java/local/flowable-lab/doc/00-总览/平台总体架构设计1.1-修订.md)。如与原文冲突，以修订说明为准。

### 1. 设计目标

本项目的目标不是直接围绕 Flowable 暴露能力，而是建设一个由平台主导的流程产品原型。

核心原则：

- 平台拥有自己的流程定义、流程版本、流程发布、基础查询模型
- Flowable 严格限制在引擎层，只负责执行能力
- 前端建模围绕平台对象展开，而不是围绕 Flowable 内部模型展开

### 2. 架构原则

#### 2.1 设计态与运行态分离

平台模型是设计态主对象。

Flowable BPMN 是平台发布后的执行产物，不是主设计对象。

#### 2.2 图主存储与逻辑骨架分离

前端主存储仍然是整份 `graph_json`。

同时，平台内部必须保留：

- 节点逻辑骨架
- 边逻辑骨架

其中：

- `graph_json` 用于前端加载、保存和画布还原
- 节点、边逻辑骨架用于平台校验、转换、挂载和路径分析

#### 2.3 外挂配置与逻辑骨架分离

节点和边只是流程图中的逻辑骨架对象，不承载复杂配置。

所有外挂配置都独立建表，由外挂配置自己挂载到具体节点或边上。

关系方向是：

- 节点候选人配置挂载到节点
- 边条件配置挂载到边

而不是：

- 节点去引用候选人配置
- 边去引用条件配置

#### 2.4 基础查询与高级查询分离

基础查询：

- 基于平台数据库读模型
- 只索引核心字段
- 支撑第一期可发起流程、待办、已办、我发起的流程

高级查询：

- 作为后续能力建设
- 基于消息中间件和搜索索引实现
- 承接复杂筛选、全文搜索和聚合分析

#### 2.5 第一迭代轻装上阵

第一期只围绕最小主链路建设：

- 流程定义
- 流程绘制
- 流程版本控制
- 流程发布
- 允许启动流程列表
- Flowable 核心能力支持
- 平台模型到 Flowable 模型转换
- 基础可读模型

外围能力、复杂运维、高级搜索统一放到后续迭代。

### 3. 总体分层

#### 3.1 设计层

负责：

- 流程定义
- 流程版本
- 图模型编辑
- 节点边逻辑骨架维护
- 节点候选人配置维护
- 边条件配置维护

#### 3.2 发布层

负责：

- 校验
- 编译
- 发布
- 部署到 Flowable
- 平台版本与引擎版本绑定

#### 3.3 引擎层

负责：

- Flowable 流程定义部署
- 流程实例启动
- 用户任务推进
- 网关执行
- 运行时与历史数据存储

#### 3.4 查询投影层

负责：

- 可发起流程列表
- 待办列表
- 已办列表
- 我发起的流程列表
- 基础实例摘要

#### 3.5 后续扩展层

后续迭代再考虑：

- 角色、部门、表达式候选人
- 复杂运维动作
- 高级搜索
- 外围业务语义治理

### 4. 第一迭代边界

#### 4.1 第一迭代支持的流程绘制能力

- 开始节点
- 结束节点
- 用户任务节点
- 排他分支节点
- 并行分支节点
- 并行汇聚节点
- 普通连线
- 条件连线
- 默认分支标记

#### 4.2 第一迭代支持的候选人能力

统一使用节点候选人配置模型，但只实现账号模式：

- 单人账号处理
- 多账号候选处理

模型上预留后续扩展位，但第一期不实现：

- 角色模式
- 部门模式
- 表达式模式

#### 4.3 第一迭代不支持的能力

- 会签与多实例
- 加签减签
- 退回跳转
- 子流程
- 定时器与消息事件
- 表达式路由
- 动态候选解析
- 发布审批
- 流程运维中心
- 高级搜索

### 5. 平台与 Flowable 的边界

平台负责：

- 流程定义与版本
- 图 JSON 主存储
- 节点和边逻辑骨架
- 节点候选人配置
- 边条件配置
- 发布记录
- 平台模型到 Flowable 的转换
- 启动入口
- 基础查询投影

Flowable 负责：

- BPMN 执行
- 流程实例运行
- 用户任务与网关推进
- 历史数据

换句话说，平台不是 Flowable 的前端壳，而是以 Flowable 为执行内核的流程平台。

### 6. 第一迭代主链路

```text
定义流程
-> 绘制流程并保存 graph_json
-> 同步节点边逻辑骨架
-> 保存草稿版本
-> 维护节点候选人和边条件配置
-> 校验
-> 发布
-> 转换为 Flowable 模型
-> 部署到 Flowable
-> 展示允许启动流程列表
-> 用户启动流程
-> 写入基础读模型
```

### 8. 结论

当前原型设计的核心策略是：

- 不把第一期做成一个重平台
- 前端仍然直接使用 `graph_json`
- 平台保留节点、边的结构化逻辑骨架
- 所有外挂配置独立建表并挂载到节点或边
- 先打通定义、版本、发布、启动、查询这条主线
- 让 Flowable 从第一天开始就被限制在引擎边界内

这套架构足以支撑后续逐步引入复杂流程能力，而不会反向被第一期模型绑死。
