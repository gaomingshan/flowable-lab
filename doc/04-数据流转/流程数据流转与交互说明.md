## 流程数据流转与交互说明

> 修订说明：本文件的后续执行口径请同时参考 [流程数据流转与交互说明 1.1 修订](C:/soft/work/code/java/local/flowable-lab/doc/04-数据流转/流程数据流转与交互说明1.1-修订.md)。如与原文冲突，以修订说明为准。

### 1. 文档目标

本文档用于说明第一迭代下，平台从流程定义、流程版本、流程发布到用户启动的核心数据流转过程。

本期只关注主链路：

- 定义流程
- 保存版本
- 发布流程
- 同步 Flowable
- 用户发起
- 进入基础查询

### 2. 参与对象

#### 2.1 前端设计器

负责：

- 节点与边编辑
- 节点候选人配置编辑
- 边条件配置编辑
- 保存图模型
- 触发校验与发布

#### 2.2 平台服务层

负责：

- 定义管理
- 版本管理
- 图 JSON 保存
- 节点边骨架同步
- 节点候选人配置管理
- 边条件配置管理
- 流程校验
- 流程编译
- Flowable 部署适配
- 启动流程
- 基础查询投影维护

#### 2.3 Flowable 引擎层

负责：

- 部署流程定义
- 启动流程实例
- 执行任务与网关
- 存储运行时和历史数据

#### 2.4 平台数据库与投影层

负责：

- 保存定义、版本、发布、绑定关系
- 保存节点边骨架
- 保存外挂配置
- 保存运行态基础投影

### 3. 主链路总览

第一迭代主链路如下：

1. 创建流程定义
2. 编辑图模型与外挂配置
3. 保存 graph_json 并同步节点边骨架
4. 保存草稿版本
5. 执行校验
6. 创建发布记录
7. 编译为 Flowable 模型
8. 发布到 Flowable
9. 写入引擎绑定关系
10. 刷新允许启动流程列表
11. 用户发起流程
12. 生成基础读模型

### 4. 定义阶段

### 4.1 创建流程定义

平台创建流程逻辑对象，写入：

- `wf_definition`

这个对象用于表达“这条流程是谁”，而不是表达具体某个可执行版本。

### 4.2 编辑图模型

前端保存：

- 节点
- 边
- 布局
- 画布元数据

这些信息写入：

- `wf_definition_version.graph_json`

### 4.3 同步节点边骨架

平台在保存图模型后，从 `graph_json` 中解析出：

- 逻辑节点
- 逻辑边

并同步写入：

- `wf_node`
- `wf_edge`

### 4.4 编辑节点候选人配置

前端通过节点候选人配置接口维护：

- 用户任务节点候选人配置

这些配置写入：

- `wf_node_candidate_config`

其中账号模式的候选人明细保存在：

- `wf_node_candidate_config.policy_payload_json`

### 4.5 编辑边条件配置

前端通过边条件配置接口维护：

- 条件连线基础条件
- 默认分支标记

这些配置写入：

- `wf_edge_condition_config`

### 4.6 保存草稿版本

平台在保存时形成版本快照，写入：

- `wf_definition_version`

该快照包括：

- 图模型快照
- 逻辑骨架同步结果
- 挂载配置结果
- 校验结果快照

### 5. 发布阶段

### 5.1 校验

平台发布前执行三类校验。

#### 5.1.1 图结构校验

例如：

- 是否存在开始节点
- 是否存在结束节点
- 是否存在孤立节点
- 网关结构是否合理

#### 5.1.2 逻辑骨架校验

例如：

- 节点 key 是否唯一
- 边 key 是否唯一
- 边的起点终点是否存在
- 某节点是否存在后继或前驱异常

#### 5.1.3 挂载配置校验

例如：

- 用户任务节点是否配置候选人
- 条件边是否缺少条件配置
- 候选人配置挂载的节点是否存在
- 边条件配置挂载的边是否存在

校验结果写入：

- `wf_definition_version.validation_report_json`
- 或 `wf_release.validation_report_json`

### 5.2 创建发布记录

平台创建：

- `wf_release`

第一期不做发布审批，因此发布记录创建后可直接进入编译和部署流程。

### 5.3 编译为 Flowable 模型

平台根据：

- `graph_json`
- `wf_node`
- `wf_edge`
- 节点候选人配置
- 边条件配置

生成 Flowable 可接受的 BPMN XML。

生成结果写入：

- `wf_release.compiled_bpmn_xml`
- `wf_release.compiled_diagram_svg`

### 5.4 部署到 Flowable

平台调用 Flowable，完成流程定义部署。

Flowable 返回：

- `deploymentId`
- `processDefinitionId`
- `processDefinitionKey`
- `processDefinitionVersion`

平台写入：

- `wf_engine_binding`

### 5.5 刷新允许启动流程列表

发布成功后，平台维护：

- `wf_launchable_projection`

用户后续从平台视角看到的是可发起流程列表，而不是直接暴露 Flowable 定义列表。

### 6. 启动阶段

### 6.1 查询允许启动流程列表

用户打开发起页时，平台读取：

- `wf_launchable_projection`

### 6.2 提交启动请求

用户提交启动请求时，平台至少接收：

- 流程定义或发布标识
- 业务 key
- 标题
- 发起人信息
- 发起部门信息
- 启动表单输入

### 6.3 定位引擎定义

平台通过以下对象定位当前应启动的 Flowable 定义：

- `wf_launchable_projection`
- `wf_release`
- `wf_engine_binding`

### 6.4 组装启动上下文

平台组装：

- 业务 key
- 标题
- 发起人
- 发起部门快照
- 启动变量

第一期只处理最基础的上下文组装，不做复杂变量治理。

### 6.5 调用 Flowable 启动实例

平台调用 Flowable 启动流程实例。

Flowable 创建：

- 流程实例
- 初始任务
- 运行时变量

### 6.6 写入实例主表与基础投影

启动成功后，平台写入：

- `wf_instance`
- `wf_todo_projection`
- `wf_initiated_projection`

任务完成后再维护：

- `wf_done_projection`

### 7. 查询阶段

### 7.1 基础查询

第一期基础查询全部走平台数据库读模型：

- 可发起流程：`wf_launchable_projection`
- 待办：`wf_todo_projection`
- 已办：`wf_done_projection`
- 我发起的：`wf_initiated_projection`
- 实例摘要：`wf_instance`

### 7.2 高级查询

高级查询不在本期落地。

后续方向是：

1. 平台运行事件投递到消息中间件
2. 搜索侧构建复杂索引
3. 支撑复杂筛选与全文搜索

### 8. 结论

第一期的数据流转核心是：

- 平台定义流程
- 平台保存版本和图 JSON
- 平台同步节点边逻辑骨架
- 平台独立维护节点候选人和边条件配置
- 平台发布并同步到 Flowable
- 平台提供启动入口
- 平台维护基础读模型

这条主链路打通后，平台就具备继续扩展复杂流程能力和高级查询能力的基础。
