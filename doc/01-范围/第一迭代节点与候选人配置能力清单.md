## 第一迭代节点与候选人配置能力清单

> 修订说明：本文件的后续执行口径请同时参考 [第一迭代节点与候选人配置能力清单 1.1 修订](C:/soft/work/code/java/local/flowable-lab/doc/01-范围/第一迭代节点与候选人配置能力清单1.1-修订.md)。如与原文冲突，以修订说明为准。

### 1. 文档目标

本文档用于明确第一迭代中允许进入平台原型的节点能力、连线能力以及候选人配置能力。

目标是两件事：

- 保证流程绘制能力足够稳定且够用
- 保证第一期不过早引入复杂 Flowable 语义

### 2. 节点能力范围

### 2.1 支持的节点类型

第一迭代建议支持以下节点。

#### 2.1.1 开始节点

用途：

- 作为流程入口

限制：

- 每个流程仅允许一个开始节点

#### 2.1.2 结束节点

用途：

- 作为流程结束点

限制：

- 一个流程允许存在多个结束节点

#### 2.1.3 用户任务节点

用途：

- 表达一个需要用户处理的审批或处理环节

第一迭代只支持基础配置：

- 节点名称
- 节点候选人配置

不支持：

- 会签
- 多实例
- 加签减签
- 催办策略
- 复杂变量配置

#### 2.1.4 排他分支节点

用途：

- 表达基于条件的单路径分支

第一迭代支持：

- 条件分支
- 默认分支

不支持：

- 复杂表达式治理
- 可视化规则编排器

#### 2.1.5 并行分支节点

用途：

- 将流程拆成并行执行路径

第一迭代只支持基础并行，不叠加会签语义。

#### 2.1.6 并行汇聚节点

用途：

- 对并行路径进行汇聚

说明：

- 第一迭代仅按 Flowable 原生并行网关能力处理

### 2.2 不支持的节点类型

第一迭代明确不支持：

- 子流程
- 调用流程
- 服务任务
- 脚本任务
- 定时器事件
- 边界事件
- 消息事件
- 信号事件
- 补偿事件
- 中间捕获/抛出事件

### 3. 连线能力范围

### 3.1 支持的连线能力

- 普通连线
- 条件连线
- 默认分支标记

### 3.2 条件连线规则

第一迭代建议先支持最基础条件能力：

- 连线上存在条件文本或条件配置
- 排他分支节点允许设置默认分支

第一期不做：

- 可视化复杂条件构建器
- 条件模板目录
- 表达式治理

### 4. 候选人配置设计

### 4.1 设计原则

第一迭代候选人必须统一抽象为节点候选人配置模型，而不是孤立字段。

这样做的目的是：

- 第一迭代只实现账号模式
- 第二迭代可以平滑扩展角色、部门、表达式模式
- 不需要推翻第一期主表结构

### 4.2 第一迭代支持的配置模型

第一期只支持：

- `policyType = user`

但支持两种模式：

- `policyMode = single`
- `policyMode = candidate`

### 4.3 候选人动态数据承载方式

第一迭代不再单独拆账号明细表，而是统一通过候选人配置中的 `policyPayload` 承载动态数据。

这样做的原因是：

- 第一迭代虽然只支持账号模式，但后续会扩角色、部门、表达式
- 如果现在就拆独立账号明细表，后续会被迫继续拆更多策略明细表
- `policyPayload` 可以在保持主模型稳定的前提下承载不同策略类型的数据结构

### 4.4 单账号处理

示例：

```json
{
  "policyType": "user",
  "policyMode": "single",
  "policyName": "固定审批人",
  "policyPayload": {
    "users": [
      {
        "userId": "u_1001",
        "userName": "张三"
      }
    ]
  }
}
```

语义：

- 当前节点由固定一个账号处理

对应 Flowable 映射方向：

- `assignee`

### 4.5 多账号候选

示例：

```json
{
  "policyType": "user",
  "policyMode": "candidate",
  "policyName": "财务候选人",
  "policyPayload": {
    "users": [
      {
        "userId": "u_1001",
        "userName": "张三"
      },
      {
        "userId": "u_1002",
        "userName": "李四"
      }
    ]
  }
}
```

语义：

- 当前节点由固定账号列表作为候选人处理

对应 Flowable 映射方向：

- `candidateUsers`

### 4.6 第一迭代不支持的配置类型

第一迭代不支持：

- `policyType = role`
- `policyType = department`
- `policyType = expression`
- `policyType = resolver`

### 4.7 第二迭代预留扩展位

虽然第一期不实现，但第一天就应预留以下扩展方向：

- `policyType = role`
- `policyType = department`
- `policyType = expression`

并通过 `policyPayload` 分别承载：

- 角色集合
- 部门集合
- 表达式配置

### 5. 节点与边的配置边界

第一迭代建议：

- 用户任务节点挂载候选人配置
- 条件边挂载条件配置
- 节点和边本身不承载复杂配置

也就是说：

- 节点和边是逻辑骨架
- 候选人和条件是外挂配置

### 6. 成功标准

如果第一期节点与候选人配置能力能支撑以下流程，就算范围设计合理：

#### 6.1 线性审批流

```text
开始 -> 用户任务 -> 用户任务 -> 结束
```

#### 6.2 条件分支流

```text
开始 -> 用户任务 -> 排他分支 -> 用户任务 / 用户任务 -> 结束
```

#### 6.3 基础并行流

```text
开始 -> 并行分支 -> 用户任务 + 用户任务 -> 并行汇聚 -> 结束
```

### 7. 结论

第一迭代的节点与候选人配置设计原则可以概括为：

- 节点能力比极简版稍宽，但仍然稳定
- 候选人统一使用节点候选人配置模型
- 第一迭代只实现账号模式
- 候选人动态数据统一放在 `policyPayload` 中
- 数据模型为第二迭代保留扩展位

这样可以既保证第一期轻装上阵，又不至于把后续扩展路线堵死。
