## 第一迭代技术选型与实现决策

### 1. 文档目标

本文档用于明确第一迭代原型的核心技术选型，以及为什么这么选。

本轮目标不是做长期终局架构，而是做第一期最小可用原型的实现决策。

### 2. 决策原则

第一期技术选型遵循四个原则：

1. 稳定优先
2. 低心智负担优先
3. 贴合当前业务模型优先
4. 不为第二期能力过度设计

### 3. 后端技术选型

### 3.1 语言与框架

建议：

- Java 17+
- Spring Boot 3.x

原因：

- 与 Flowable 集成自然
- Spring 生态成熟
- 配置、事务、校验、依赖管理完整

不建议：

- 第一迭代通过 Node.js 或 Go 直接包一层 Flowable
- 第一迭代拆微服务

### 3.2 Web API 风格

建议：

- REST 风格 API
- JSON 请求与响应

原因：

- 适合前端原型阶段联调
- 文档和接口边界清晰

### 3.3 数据访问层

建议：

- MyBatis 或 MyBatis-Plus

原因：

- 对平台自定义表结构控制更直接
- 对读模型、投影表、自定义查询更友好
- 相比 JPA，更容易明确 SQL 行为和索引路径

### 3.4 参数校验与数据转换

建议：

- Spring Validation 做请求校验
- DTO / DO / Query Object 分层

原因：

- 第一期就建立边界
- 避免平台模型直接暴露数据库对象

### 4. 数据库选型

建议：

- MySQL 8.x 或 PostgreSQL 14+

第一期如果没有额外约束，优先建议：

- MySQL 8.x

原因：

- 团队普遍熟悉度高
- 与 Flowable 默认使用习惯兼容度高
- 足够支撑第一期平台表、投影表和 JSON 字段存储

### 5. 前端技术选型

### 5.1 前端框架

建议：

- React + TypeScript

原因：

- 节点图编辑生态成熟
- 图编辑、状态管理、动态配置面板实现方便
- TypeScript 对流程图 JSON 协议和配置结构约束更有价值

如果现有团队 Vue 更强，也可以落 Vue 3 + TypeScript，但需要统一团队方向。

### 5.2 图编辑库

建议：

- React Flow / XYFlow 类成熟图编辑库

原因：

- 直接满足节点、边、拖拽、缩放、选择、连线等需求
- 不需要第一期自研底层画布能力
- 与 `graph_json` 结构天然匹配

不建议：

- 第一迭代自研底层节点图框架
- 直接使用 BPMN 传统编辑器作为前端主交互

### 5.3 状态管理方式

建议：

- 前端图状态与配置状态分开管理

图状态：

- 当前版本 `graph_json`

配置状态：

- 节点候选人配置
- 边条件配置

原因：

- 与平台后端模型一致
- 减少整图重保存时的副作用

### 6. 图模型实现决策

### 6.1 主存储对象

建议：

- 前端保存整份 `graph_json`
- 后端版本表保存 `graph_json`

### 6.2 骨架同步

建议：

- 每次保存图后，后端立即解析并同步 `wf_node`、`wf_edge`

原因：

- 后端后续逻辑不必每次都重新解析 JSON
- 节点和边的后继、前驱、校验和转换会更清晰

### 6.3 第一迭代不建议的方式

不建议：

- 只存 `graph_json` 而完全不维护 `wf_node`、`wf_edge`
- 节点和边再额外存复杂 JSON 配置

### 7. 配置模型实现决策

### 7.1 节点候选人配置

建议：

- 独立表 `wf_node_candidate_config`
- 动态策略数据放 `policy_payload_json`

原因：

- 第一期支持账号模式足够轻
- 第二期扩角色、部门、表达式不推翻主结构

### 7.2 边条件配置

建议：

- 独立表 `wf_edge_condition_config`

原因：

- 边条件是第一期最核心的边外挂配置
- 单独建模比挂进图 JSON 更利于校验和转换

### 8. 查询模型实现决策

### 8.1 基础查询

建议：

- 通过数据库投影表实现

包括：

- `wf_launchable_projection`
- `wf_todo_projection`
- `wf_done_projection`
- `wf_initiated_projection`

原因：

- 第一迭代要先保证页面可用
- 避免直接耦合 Flowable 查询模型

### 8.2 高级查询

建议：

- 明确不进入第一期实现

原因：

- 容易把原型拉重
- 需要消息与搜索体系配合，不适合与主链路一起做

### 9. 第一迭代项目结构建议

建议：

- 单仓库
- 单后端应用
- 单前端应用

后端内部按模块分包：

- definition
- version
- graph
- candidate
- condition
- release
- runtime
- query
- flowable adapter

原因：

- 保持项目结构清晰
- 又不引入第一期不必要的分布式复杂度

### 10. 结论

第一期技术选型建议可以收敛为：

- Java 17 + Spring Boot 3
- MyBatis / MyBatis-Plus
- MySQL 8.x
- React + TypeScript
- React Flow / 类似成熟图编辑库
- `graph_json` 主存储 + `wf_node/wf_edge` 逻辑骨架同步
- 独立外挂配置表
- 数据库基础读模型

这套组合最适合当前的第一迭代目标：轻、稳、可打通主链路。
