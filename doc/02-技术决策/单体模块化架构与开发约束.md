## 单体模块化架构与开发约束

### 1. 文档目标

本文档用于明确第一迭代在技术架构、模块拆分、开发风格和基础组件选型上的统一约束。

这份文档不回答“业务做什么”，而回答：

- 代码如何组织
- 模块如何交互
- 哪些技术路线允许采用
- 哪些技术路线明确禁止

本文档作为第一迭代的研发执行约束，优先级高于普通实现习惯。

### 2. 总体技术立场

第一迭代采用以下总体技术立场：

1. 单体优先，不拆分布式服务
2. 模块化优先，不写大泥球单体
3. 适度采用 DDD 思想，不走重型 DDD
4. 接口优先，不允许跨模块直接依赖实现
5. 事件优先，用于模块间和模块内解耦
6. 成熟轮子优先，禁止手搓基础能力

### 3. DDD 采用深度

#### 3.1 采用什么

第一迭代采用 DDD 的以下思想：

- 以领域边界划分模块
- 使用统一领域语言命名对象和接口
- 区分应用编排与领域对象
- 避免业务语义被技术分层冲散

#### 3.2 不采用什么

第一迭代不采用重型 DDD 套餐，不追求：

- 到处都是聚合根
- 所有对象都包装成值对象
- 仓储模式过度泛化
- 领域服务泛滥
- 复杂领域基础设施抽象

#### 3.3 当前推荐理解

第一迭代的 DDD 深度应该理解为：

- 模块边界按业务划分
- 重要概念按领域命名
- 关键逻辑不要散落在控制器和 Mapper 中

这已经足够支撑第一期原型。

### 4. 模块化单体架构约束

#### 4.1 架构风格

建议采用：

- Spring Boot 单体应用
- 模块化单体架构
- 以 Spring Modulith 思想组织模块

第一迭代不要求必须完整引入 Spring Modulith 全套功能，但模块划分和交互方式应尽量与其理念一致。

#### 4.2 模块拆分原则

模块按业务职责拆分，而不是按纯技术分层拆分。

建议第一迭代模块包括：

- `workflow.definition`
- `workflow.version`
- `workflow.graph`
- `workflow.candidate`
- `workflow.condition`
- `workflow.release`
- `workflow.runtime`
- `workflow.query`
- `workflow.flowable`

#### 4.3 禁止的模块组织方式

禁止整个项目按全局技术层分包，例如：

- `controller`
- `service`
- `mapper`
- `entity`

这种组织方式会迅速削弱模块边界。

### 5. 模块内推荐结构

每个模块内部建议拆成以下层次：

- `api`
- `application`
- `domain`
- `infrastructure`

#### 5.1 `api`

负责：

- 对外暴露接口
- command / query DTO
- 对外事件定义

#### 5.2 `application`

负责：

- 用例编排
- 事务控制
- 调用领域对象和模块接口

#### 5.3 `domain`

负责：

- 领域对象
- 领域规则
- 不依赖外部基础设施的核心逻辑

#### 5.4 `infrastructure`

负责：

- 数据访问实现
- Flowable 适配实现
- 外部系统接入实现

### 6. 接口依赖约束

#### 6.1 对外只暴露接口

模块对外只能暴露：

- facade interface
- command / query object
- integration event

#### 6.2 跨模块禁止依赖实现类

跨模块只能依赖接口，禁止：

- 直接 `new` 其他模块实现类
- 直接注入其他模块的实现类
- 直接引用其他模块的内部包

#### 6.3 Flowable 依赖边界

除 `workflow.flowable` 模块及必要的应用编排层之外，其他模块不得直接依赖 Flowable API。

也就是说：

- 平台定义模块不应直接操作 Flowable
- 查询模块不应直接查询 Flowable 原表
- 候选人模块不应依赖 Flowable 类型

### 7. 事件驱动约束

#### 7.1 模块间交互

模块间交互优先通过：

- 集成事件
- 对外接口

推荐原则：

- 查询型能力通过接口调用
- 状态传播和解耦通过集成事件

#### 7.2 模块内交互

模块内也允许使用事件进行解耦。

适用场景：

- 发布成功后触发模块内附加动作
- 启动流程后刷新本模块状态
- 编译完成后触发文档/快照更新

#### 7.3 第一迭代事件实现方式

第一期建议：

- 使用 Spring 应用事件机制
- 先走同进程事件
- 不将 MQ 作为主链路依赖

#### 7.4 事件命名建议

推荐使用显式领域事件名称，例如：

- `WorkflowReleasedEvent`
- `WorkflowStartedEvent`
- `TodoProjectionRefreshedEvent`

不要使用无语义的通用事件名称。

### 8. JSON 使用边界

#### 8.1 可以使用 JSON 的地方

第一期明确允许使用 JSON 的地方：

- `graph_json`
- 候选人 `policy_payload_json`
- 启动页元数据
- 简单校验报告
- 候选人快照

#### 8.2 禁止的 JSON 使用方式

禁止：

- 把整个平台外围能力都塞进一个万能配置 JSON
- 用 JSON 代替本该结构化的定义、版本、发布关系
- 把 Flowable 编译逻辑完全建立在未约束的 JSON 结构上

#### 8.3 判断标准

如果某个对象：

- 关系稳定
- 查询频繁
- 生命周期明确

优先结构化。

如果某个对象：

- 动态结构明显
- 第一迭代仅需轻量支持

可以先 JSON 化。

### 9. Lombok 使用约束

#### 9.1 允许使用

建议优先使用：

- `@Getter`
- `@Setter`
- `@Builder`
- `@RequiredArgsConstructor`
- `@NoArgsConstructor`
- `@AllArgsConstructor`

#### 9.2 不建议滥用

不建议：

- 全项目无脑使用 `@Data`

原因：

- 容易模糊对象边界
- `equals/hashCode/toString` 不受控

#### 9.3 使用范围

Lombok 主要用于：

- DTO
- Command / Query Object
- Persistence Object
- 简单领域载体对象

### 10. MapStruct 使用约束

#### 10.1 统一采用

对象转换统一建议采用 MapStruct，不鼓励手写大段重复转换代码。

#### 10.2 适用场景

包括：

- DTO -> Command
- Command -> Entity
- Entity -> Response DTO
- Persistence Object -> Domain Object

#### 10.3 不建议的方式

不建议：

- 在 Controller 中手写大段字段拷贝
- 到处使用通用 BeanUtils 做不透明转换

### 11. 成熟组件优先，禁止手搓

#### 11.1 明确优先复用的能力

第一迭代优先复用：

- Flowable：流程执行引擎
- Spring Boot：应用基础设施
- Spring 事件机制：模块事件
- MapStruct：对象转换
- Lombok：样板代码简化
- Bean Validation：参数校验
- Vue 3 生态图编辑库：节点图能力

#### 11.2 明确禁止手搓的范围

禁止自研这些底层能力：

- 流程执行引擎
- 节点图底层引擎
- DTO 映射基础设施
- 分页基础设施
- 模块边界治理框架

### 12. 前端技术约束

#### 12.1 前端框架

第一迭代前端统一采用：

- Vue 3
- TypeScript

#### 12.2 状态管理

建议：

- 图状态与配置状态分离管理

图状态：

- `graph_json`

配置状态：

- 节点候选人配置
- 边条件配置

#### 12.3 图编辑能力

建议：

- 使用成熟 Vue 生态节点图方案
- 不自研底层画布协议

#### 12.4 前端禁令

禁止：

- 前端直接拼装 Flowable BPMN XML
- 前端直接依赖 Flowable 内部模型
- 前端自己维护平台主逻辑关系而后端不落骨架

### 13. 测试与验证约束

#### 13.1 后端测试重点

第一期至少要覆盖：

- 节点边骨架同步
- 候选人配置解析
- 边条件配置解析
- 平台模型到 Flowable 模型转换
- 基础读模型同步

#### 13.2 模块测试重点

建议逐步形成模块级测试习惯，优先覆盖：

- 发布主链路
- 启动主链路
- 查询投影刷新链路

### 14. 当前执行口径

如果现在开始研发，统一按以下技术口径执行：

1. 适度 DDD，按模块组织代码
2. 模块间优先接口和集成事件
3. 模块内允许通过事件解耦
4. 禁止跨模块直接依赖实现
5. 统一采用 Lombok + MapStruct
6. 优先成熟组件，禁止手搓基础能力
7. 前端采用 Vue 3 + TypeScript

### 15. 结论

这份文档的作用，是把“可以怎么写代码”从个人习惯变成团队约束。

第一迭代只要严格执行这些约束，就能在单体架构下保持模块高内聚、低耦合，为第二迭代继续演进保留足够空间。
