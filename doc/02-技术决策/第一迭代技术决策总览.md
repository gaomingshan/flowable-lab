## 第一迭代技术决策总览

### 1. 文档目标

本文档用于承接当前已经基本稳定的业务设计，进入第一迭代的技术决策阶段。

业务设计主要回答：

- 要做什么
- 边界是什么
- 平台模型如何定义

技术决策主要回答：

- 第一迭代如何实现
- 用什么技术栈
- 为什么这么选
- 哪些实现方式明确不选

### 2. 当前技术决策范围

第一迭代技术决策只围绕当前主链路展开：

- 流程定义保存
- 流程图 JSON 保存
- 节点边逻辑骨架同步
- 节点候选人配置
- 边条件配置
- 平台模型到 Flowable 模型转换
- 发布到 Flowable
- 启动流程
- 基础读模型维护

不进入当前技术决策范围的内容：

- 高级搜索引擎选型
- 消息中间件最终接入方案
- 运维动作中心实现
- 动态候选人解析框架
- 复杂表达式治理框架
- 多实例与会签引擎扩展

### 3. 第一迭代技术决策主题

第一期建议先锁定以下五类技术决策：

#### 3.1 后端技术栈

需要明确：

- 语言与框架
- ORM 或数据访问方式
- API 风格
- 校验方式

#### 3.2 前端技术栈

需要明确：

- 图编辑库
- 状态管理方式
- 配置页签实现方式
- JSON 协议约束方式

#### 3.3 Flowable 集成方式

需要明确：

- 是否嵌入式集成
- 是否通过 REST 调用
- 编译与部署边界
- 启动和任务查询边界

#### 3.4 数据存储策略

需要明确：

- `graph_json` 存储方式
- 节点边骨架同步方式
- 候选人 payload 存储方式
- 基础读模型更新方式

#### 3.5 模块拆分方式

需要明确：

- 第一迭代是否拆成多模块
- 平台服务与 Flowable 适配是否独立
- 前端与后端的原型分界

### 4. 当前建议的技术决策顺序

建议按以下顺序推进：

1. 后端基础技术栈
2. Flowable 集成方式
3. 数据存储与同步策略
4. 前端图编辑与配置交互实现方式
5. 模块拆分与项目结构

### 5. 当前默认技术方向

在没有进一步约束之前，我建议第一迭代优先采用稳定、低心智负担的路线：

#### 5.1 后端

- Java
- Spring Boot
- MySQL 或 PostgreSQL
- MyBatis-Plus 或 MyBatis 风格数据访问
- Spring Validation

#### 5.2 Flowable 集成

- 同进程嵌入式集成 Flowable Engine
- 平台服务直接调用 Flowable Java API
- 不走 Flowable REST 作为第一迭代主路径

#### 5.3 前端

- TypeScript
- React 或 Vue 二选一
- 节点图编辑库优先使用成熟图编辑库
- 不自研底层画布协议

#### 5.4 数据策略

- `graph_json` 作为主存储
- 每次保存图后同步 `wf_node`、`wf_edge`
- 候选人动态数据存入 `policy_payload_json`
- 基础读模型通过平台同步写入维护

### 6. 当前不建议的实现方向

第一迭代明确不建议：

- 前端直接对接 Flowable REST 建模和发布
- 完全依赖 `graph_json` 做所有运行逻辑判断
- 把所有外围配置塞进一个万能扩展 JSON 表
- 引入过早的事件驱动、消息最终一致性主链路
- 同时做复杂搜索引擎和基础读模型

### 7. 与业务设计的关系

当前技术决策必须服从以下业务设计结论：

1. `graph_json` 是前端主存储
2. `wf_node`、`wf_edge` 是轻量逻辑骨架
3. 外挂配置独立建表并直接挂载到节点或边
4. 候选人主配置结构化，策略动态数据 JSON 化
5. Flowable 只在引擎层承担执行职责
